# Investment OS 로직 분석 및 리팩토링 검토 (2026-04-18)

## 1) 현재 메인 로직 분석

### A. 시장 자료 수집
- `run_market.py`의 Step 1~2에서 야후파이낸스/연준(FRED)/RSS/크립토/Fear&Greed/유튜브 RSS 데이터를 순차 수집합니다.
- 수집 실패를 대부분 `warning`으로 처리해 파이프라인을 계속 진행하도록 설계되어 있습니다(가용성 우선).

### B. 내용 요약 및 대시보드 생성
- `run_market.py`에서 뉴스 요약(`collectors/news_summarizer.py`) 및 Gemini 기반 심층 분석(`collectors/news_analyzer.py`)을 수행합니다.
- `run_view.py`에서 텍스트 포맷팅 후 이미지 생성(`publishers/image_generator.py`)을 호출해 대시보드 이미지를 생성합니다.

### C. 유명 유튜버 의견 취합 요약
- `collectors/youtube_rss.py`가 채널별 RSS를 수집하고 최근 영상을 추립니다.
- `engines/streamer_analyzer.py`가 Gemini로 영상 제목/설명을 통합 요약해 단일 의견/트윗 형태로 생성합니다.
- `run_view.py`에서 morning 세션 시 유튜버 요약 트윗을 별도 발행합니다.

### D. X/텔레그램 발행
- `run_view.py`가 Validation → 중복검사 → 게시글 생성 → X 발행 → 텔레그램 발행 순서로 진행합니다.
- 단일 트윗/스레드/이미지+스레드 조합까지 처리하며, weekly/full 세션의 텔레그램 분기 로직도 포함합니다.

## 2) 구조적 문제점 (우선순위 포함)

## P0 (우선 해결 권장)
1. **중복 엔트리/모듈 공존으로 인한 유지보수 리스크**
   - 루트와 패키지 하위에 동일한 파일명(`run_view.py`, `risk_engine.py`, `x_publisher.py` 등)이 다수 공존합니다.
   - 실제로 `publishers/run_view.py`와 루트 `run_view.py`는 내용이 크게 다르고 기능 버전도 다릅니다.
   - 영향: 잘못된 import/실행 경로로 구버전 로직이 호출될 가능성, 버그 재발 가능성 증가.

2. **파이프라인 함수의 과도한 비대화**
   - `run_market.py`와 `run_view.py`가 단일 함수(`run`)에 너무 많은 책임(수집/분석/포맷/발행/알림/예외처리)을 가지고 있습니다.
   - 영향: 테스트 어려움, 변경 시 회귀 위험 증가, 장애 원인 추적 어려움.

## P1 (단기 개선 권장)
3. **예외 처리 기준 불균일**
   - 일부는 Hard Gate(Validation FAIL 시 즉시 중단), 다수는 soft fail(경고 후 진행)로 섞여 있으나 기준이 코드상 선언적으로 정리되어 있지 않습니다.
   - 영향: 운영자가 "무엇이 실패해도 계속 가는지"를 빠르게 파악하기 어려움.

4. **문서/버전 정보 불일치 가능성**
   - README의 버전과 코드 내 버전 주석/설정 상수가 상이하게 보입니다.
   - 영향: 운영 중 현재 배포 버전 혼동, 트러블슈팅 속도 저하.

5. **스케줄 기준 시간대와 미국시장 운영관점의 분리 필요**
   - 스케줄러와 세션 감지는 KST 중심입니다.
   - 미국 투자 판단 서비스라면 ET 기준 운영 모드가 선택 가능해야 운영/검증이 더 직관적입니다.

## P2 (중기 개선 권장)
6. **도메인 로직과 채널별 출력 로직 결합도 높음**
   - 투자 판단(시그널) 로직과 X/텔레그램 표현 로직이 파이프라인에서 밀접히 얽혀 있습니다.
   - 영향: 신규 채널 추가(예: 웹 대시보드 API) 시 영향 범위가 큼.

7. **외부 API/AI 의존 로직의 관측성(Observability) 보강 필요**
   - 실패는 로깅되지만 지표화(실패율, 응답시간, fallback 사용률) 축적 구조가 약합니다.
   - 영향: 장기적인 품질관리/비용관리 난이도 증가.

## 3) 고도화/리팩토링 제안

### 3-1. 아키텍처 정리 (강력 권장)
- 루트 레벨의 중복 모듈을 정리하고, **단일 패키지 네임스페이스**(예: `investment_os/`)로 통합.
- 실행 엔트리는 `main.py` 하나로 고정하고, 내부에서만 명시적 import 경로 사용.
- "현재 사용 중"과 "레거시 보관" 파일을 물리적으로 분리(`legacy/` 이동 또는 삭제).

### 3-2. 파이프라인 분해
- `run_market.run()`을 단계별 유스케이스 함수로 분리:
  - `collect_inputs()`
  - `compute_signals()`
  - `assemble_core_payload()`
  - `persist_outputs()`
- `run_view.run()`도 분해:
  - `validate_payload()`
  - `build_posts()`
  - `publish_x()`
  - `publish_telegram()`
  - `record_history()`

### 3-3. 실패 정책 선언화
- 데이터 소스별 실패 정책을 코드상 상수로 관리:
  - `REQUIRED_SOURCES = {...}`
  - `OPTIONAL_SOURCES = {...}`
- 결과 리포트에 `degraded_mode: true/false`를 명시해 운영자가 한눈에 상태를 파악.

### 3-4. 테스트 전략 개선
- 현재 테스트 외에 **계약 테스트(contract tests)** 추가:
  - 수집기 출력 스키마 고정 테스트
  - 발행 payload 스키마 테스트
- 파이프라인 E2E는 외부 API mock 버전과 live smoke 버전 분리.

### 3-5. 미국 투자 서비스 관점 기능 강화
- ET 기준 세션(프리마켓/정규장/애프터마켓) 옵션 추가.
- 핵심 리스크 지표의 "장중 변동 트리거"를 별도 실시간 경보로 분리.
- 유튜버 의견은 품질 안정화를 위해 채널 신뢰도 가중치/주제 분류(금리, AI, 매크로) 추가.

4. README/설계문서 버전 일치화 자동 체크(간단한 CI 스크립트)
